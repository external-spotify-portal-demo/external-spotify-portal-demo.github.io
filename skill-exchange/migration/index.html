Migrating from previous versions

As of version 0.9.0, Skill Exchange no longer requires a Docker service backend to operate, greatly simplifying the installation and maintenance of the plugin. The Docker backend service is now deprecated, and support will cease with the next version of Skill Exchange plugins.

If this is the first time you're installing the Skill Exchange plugin, then it is strongly recommended you use the new backend service by following the steps above, and ignore this section.

For existing users, the default now is to use the new Skill Exchange backend plugin, and we have provided migration steps below to make the switch. However, users who want to opt out of this update can include the following configuration in their app-config.yaml (or equivalent) file.

skillExchange:
  apiHost: http://localhost:8080
+  disableNewApi: true

Migrating from the legacy Docker backend service​

If this is your first time installing the Skill Exchange plugin, you can safely ignore this section. However, if you are running Skill Exchange with the Docker backend service and wish to migrate your data to the new Backstage backend plugin, you can do so by following the steps below.

Prerequisites​

It's highly recommended that you back up your data before starting the migration. While the migration does not delete any previous data, we still recommend backing up any sensitive data in case of unforeseen failure.

1. Configuring the Skill Exchange Plugin database connection​

By default, the Skill Exchange backend plugin will create a new database when started up. Therefore, we must configure the plugin to connect to the previously created database. Depending on how you set up the Docker backend service, you can do this by either passing the full connection URL via a per-plugin configuration in your app-config.yaml (or equivalent) file, as shown below:

backend:
  database:
    # ... Existing database configuration
+   plugin:
+     skillEx:
+       connection: 'postgresql://existing.pg-instance.tld:5432/skill-exchange-db'

or if you used the same database server for the Docker and Backstage backend services, then you can just set the name of the Skill Exchange database to what was used by the Docker service:

backend:
  database:
    # ... Existing database configuration
+   plugin:
+     skillEx:
+       connection:
+         database: skill-exchange-db

More information on this can be found in the Backstage Database Plugin documentation.

2. Start-up the Skill Exchange plugin​

Now that we have configured the connection, we can start the Skill Exchange plugin, which will automatically run the migration scripts to set up the new database schema and migrate any existing data. To do this, start Backstage normally.

In the logs you should see similar output to below, which will confirm the migration script ran

3. [Optional] Fully decommissioning the Docker Service containers​

Once you’ve followed the migration instructions, you’ll be using the new Skill Exchange backend service and should see all your data migrated to the new schema.

However, if you used a separate database server for the Docker service, you can only partially decommission those containers as the database server is still in use by the Skill Exchange plugin. This might not be desirable for some.

To resolve this, we can use pg_dump to back up and pg_restore to copy the migrated data from the previous database server to the one used by the rest of the Backstage plugins.

Remove per-plugin database configuration​

The first step is to remove the per-plugin database configuration from your app-config.yaml (or equivalent) that we added in the migration instructions. This will configure the Skill Exchange to use the default database connection, creating an empty plugin database alongside the other plugins.

Also, ensure any skill ingestion configuration is disabled which could lead to conflicts when copying the skill data from the previous database.

backend:
  database:
    # ... Existing database configuration
-   plugin:
-     skillEx:
-       connection: 'postgresql://existing.pg-instance.tld:5432/skill-exchange-db'

skillExchange:
-  skillIngestion:
-    location: /path/to/default-skills.yaml

After restarting Backstage, you can confirm this step was successful by going to the Skill Exchange plugin, where there should be no posts. Be sure to take note of the name of the database, as we’ll need that in the next step. It should be something like backstage_plugin_skillEx

Backup and Restore the Data​

We will use the pg_dump and pg_restore command-line tools to create a backup of the data we just migrated from the Docker Service database and restore it to the empty database we created in the previous step.

Using the command below, we can do that in one go. You’ll be prompted twice for passwords — first, for the Docker Service database password, then the Backstage Plugins database password.

We must also modify the command below to make it work with your setup. The parts that need modifying are:

PropertyDescription
DOCKER_DB_USERUsername of the Docker Service database. E.g. postgres
DOCKER_DB_HOSTThe IP of the Docker Service database. E.g. _127.0.0.1
DOCKER_DB_PORTThe port of the Docker Service database. E.g. 5432
DOCKER_DB_NAMEThe name of the Docker Service database containing the data we want to move. E.g. postgres
BS_DB_USERUsername of the Backstage Plugins database
BS_DB_HOSTThe IP of the Backstage Plugins database
BS_DB_PORTThe port of the Backstage Plugins database
BS_DB_NAMEThe name of the Skill Exchange plugin database, which we took note of in step 1. E.g. backstage_plugin_skillEx

Once you’ve replaced the placeholder in the command below with the appropriate values, you can run the command from a terminal to copy the data.

pg_dump -U <DOCKER_DB_USER> -h <DOCKER_DB_HOST> -p <DOCKER_DB_PORT> -d <DOCKER_DB_NAME> --data-only -Fc \
--exclude-table=category \
--exclude-table=event \
--exclude-table=vote_session \
--exclude-table=vote \
--exclude-table=comments \
--exclude-table=hacks \
--exclude-table=media \
--exclude-table=hack_category \
--exclude-table=hack_following \
--exclude-table=skill \
--exclude-table=connections \
--exclude-table=saved_search \
--exclude-table=bookmarks \
--exclude-table=self_reported_skill \
--exclude-table=skill_exchange_post \
--exclude-table=flyway_schema_history \
--exclude-table=backstage_backend_tasks__knex_migrations \
--exclude-table=backstage_backend_tasks__knex_migrations_lock \
--exclude-table=flyway_schema_history \
--exclude-table=knex_migrations \
--exclude-table=knex_migrations_lock \
--exclude-table=skill_exchange__tasks \
--exclude-table=backstage_backend_tasks__tasks \
--exclude-table=comments_id_seq \
--exclude-table=hacks_id_seq \
--exclude-table=media_id_seq \
--exclude-table=saved_search_id_seq \
--exclude-table=skill_exchange_post_id_seq \
--exclude-table=skill_id_new_seq \
--exclude-table=vote_session_id_seq \
| pg_restore -U <BS_DB_USER> -h <BS_DB_HOST> -p <BS_DB_PORT> -d <BS_DB_NAME> --data-only --disable-triggers -Fc

You can confirm the success of the restoration by going into the Skill Exchange plugin and checking that all your data is now visible. Once you’re happy that the data was copied correctly, you can stop all Docker service's associated containers.
